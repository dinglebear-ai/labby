## Key Findings

- `write_gateway_config` (`config.rs:34`) calls `validate_upstreams` → `validate_upstream` → `validate_gateway_url` on **every** config write, not just `gateway.add`. The SSRF guard therefore applies to all write paths including reload.
- `seed_config` (`manager.rs:141`) bypasses validation by writing directly to `Arc<RwLock<LabConfig>>` — used only in tests to inject pre-existing state.
- `validate_registry_url` in `cli/mcpregistry.rs:199` uses `ToSocketAddrs` (synchronous DNS) which blocks the thread — calling it from async without `spawn_blocking` starves the executor.
- The mcpregistry API uses cursor-based pagination: `metadata.nextCursor` is a string token, not an integer offset. The catalog had declared `offset: integer` (`catalog.rs`) which would mislead callers.
- IPv6 SSRF coverage requires three separate checks: ULA `fc00::/7` (`s[0] & 0xfe00 == 0xfc00`), link-local `fe80::/10` (`s[0] & 0xffc0 == 0xfe80`), and IPv4-mapped `::ffff:0:0/96` (`s[0..5] == 0, s[5] == 0xffff`), plus a recursive v4 check on the embedded address.

## Technical Decisions

- **SSRF guard placed in `validate_gateway_url`** (called from all write paths) rather than only the `gateway.add` dispatch arm. This means reloading a config file that already contains an RFC1918 URL also fails — intentional for homelab threat model (existing bad entries should not silently persist).
- **`spawn_blocking` at the call site only** — `validate_registry_url` remains a sync `fn`. Keeping it sync lets it be used in unit tests and non-async contexts; the blocking isolation is the caller's responsibility (documented via `# MUST NOT call from async context without spawn_blocking`).
- **Double `?` unwrap** for `spawn_blocking`: outer `?` propagates `JoinError` (task panicked), inner `?` propagates the validation `Result`. Both carry distinct `.context()` messages.
- **Fixture URL `https://fixture.example.com`** chosen for updated tests — uses an IANA-reserved example domain, HTTPS, and a non-routable hostname, so no real DNS lookup occurs and the test intent (testing behavior, not the specific URL) is preserved.
- **`validate_registry_url` in CLI kept separate from `validate_gateway_url` in dispatch** — the CLI version uses `anyhow::Result`, the dispatch version uses `ToolError`. They implement the same logic independently to avoid a cross-layer dependency.

## Risks and Rollback

- **Breaking change for homelab HTTP gateways**: Users with existing `http://` URLs in their gateway config will see `write_gateway_config` fail on reload. This is intentional per the bead's threat model, but is a silent breaking change for any such existing configs. Rollback: revert commit `59275b8` to restore permissive URL validation.
- **DNS TOCTOU**: The SSRF guard uses synchronous DNS resolution at add/write time. A short-TTL DNS rebind attack could bypass the guard after validation. This is acknowledged as acceptable for the homelab threat model (noted in inline comment at `cli/mcpregistry.rs:129`).

## Decisions Not Taken

- **Scope `validate_gateway_url` SSRF check to `gateway.add` only** (move it out of `validate_upstream`): Rejected because `write_gateway_config` is the integrity gate for the config file. An RFC1918 URL that snuck in via a direct file edit would silently persist through reloads if validation only ran on the add path.
- **Make `validate_registry_url` async** (use `tokio::net::lookup_host`): Rejected — the function is also called in sync unit tests. Keeping it sync and wrapping the async call site with `spawn_blocking` is the correct Tokio pattern for this case.
